# Pull Request Guidelines

## Title Format

PR titles must follow [Conventional Commits](https://www.conventionalcommits.org/):

```
<type>: <short description>
```

**Types:** `feat`, `fix`, `docs`, `refactor`, `test`, `chore`, `perf`, `security`, `infra`

**Examples:**
```
feat: add bulk stage command for change requests
fix: handle token refresh race condition
docs: add change request workflow documentation
```

## PR Checklist

- [ ] Title follows conventional commit format
- [ ] `make fmt && make vet && make test && make build` all pass
- [ ] New features have tests
- [ ] Breaking changes are documented
- [ ] CHANGELOG is updated (auto-generated by Release Please)

## Size Guidelines

- Keep PRs focused on a single concern
- Large refactors should be split into logical, reviewable steps
- Prefer multiple small PRs over one large PR

## Review Process

- All PRs require at least one approval
- CI must pass before merge
- Squash merge is preferred for clean history
